# Runtime Roadmap

This document tracks the work needed to make Norscode run without C and without a separate interpreter in the runtime path.

## Status

- Phase 1: completed
- Phase 2: completed
- Phase 3: completed
- Phase 4: completed
- Phase 5: completed
- Phase 6: completed

## Goal

- Native `norscode` binary
- No `clang` in the normal run path
- No separate interpreter requirement for end users
- Bytecode as the stable execution contract

## Phase 2 Notes

The bytecode backend validates `norscode-bytecode-v1` payloads before running
them, and the generated bytecode for `tests/test_math.no` still matches the
checked-in `tests/test_math.ncb.json` fixture.

The native runtime project now exists under `runtime/` with opcode, operand,
value, error, loader, stdio, and VM skeleton modules. The loader can read
`norscode-bytecode-v1` payloads and turn them into structured runtime data,
including optional globals. The loader now also rejects duplicate labels and
missing jump targets before execution starts. It also verifies that each
function key matches its declared module and name, and duplicate import aliases
before execution starts. It also rejects duplicate or empty function
parameters and empty function/import metadata. It also rejects empty labels,
empty entry values, empty global names, and bytecode payloads without functions
before execution starts. The VM skeleton now includes stack helpers, name
lookup fallback to globals, a native `PRINT` opcode, and explicit runtime
arity checks for function calls. The VM also rejects negative list-build and
call arities before they can be turned into oversized `usize` values.

## Phase 3 Notes

The native VM skeleton now has stack helpers, stack manipulation opcodes,
arithmetic and comparison ops, label-based jumps, call/return, global fallback
for names, and `PRINT`/`HALT` behavior. The VM tests now also include a direct
label-jump regression plus a `JUMP_IF_FALSE` regression so control-flow
changes get caught early, a matching `JUMP_IF_FALSE` fallthrough regression,
and a call-frame isolation regression so local state stays in the right frame.
There is also a direct `step()`-without-frames regression so the VM lifecycle
boundary stays explicit, plus an `invoke_function()` unknown-function
regression so missing call targets fail cleanly. `RET` without an explicit
value is also locked to `Null` so return behavior stays predictable.

## Phase 5: Remove the old dependencies from runtime

- Native-first runtime entrypoints are in place on the runtime bridge side.
- Those entrypoints now try to build the native runtime automatically before they run, and they report an error if the binary is still missing.
- The runtime suite and compare suite JSON now carry `build_output` at the top level when that automatic build actually happens.
- `runtime-test` JSON also carries top-level `build_output` when auto-build actually happens.
- `norscode test` JSON also carries top-level `build_output` when the suite auto-builds native runtime.
- `norscode test`, `runtime-suite`, `runtime-compare-suite`, `runtime-verify`, `runtime-smoke`, `runtime-compare-verify`, and `runtime-ci` JSON also surface `fallback_used` and `fallback_used_count` so fallback use is visible in the report.
- `runtime-test` and `norscode test <file> --json` also surface `fallback_used_count` on the single-result payload.
- `runtime-verify`, `runtime-smoke`, and `runtime-compare-verify` JSON also expose `fallback_used` and `fallback_used_count` at top level for the single build-first run.
- Those same build-first JSON outputs also expose `runtime_backend` at top level.
- `runtime-compare` JSON also surfaces `fallback_used` and `fallback_used_count` on the individual result whenever fallback is actually used.
- `norscode run --engine python --json` also surfaces `fallback_used` and `fallback_used_count` on top level.
- `runtime-run` and `runtime-check` are the clearest native-runtime bridge paths in this workspace snapshot.
- `bytecode-run` is now strict native and no longer falls back to bytecode when native runtime is missing.
- `runtime-compare` JSON also surfaces `fallback_used_count` on the result payload.
- Those same JSON outputs also surface `runtime_backend`, so fallback vs native is easy to detect in scripts.
- `runtime-ci` JSON now also carries top-level `build_output` when auto-build actually happens.
- `runtime-verify`, `runtime-smoke`, and `runtime-compare-verify` JSON also carry top-level `build_output` when auto-build actually happens; if you pass `--runtime-binary`, that explicit binary is used without fallback.
- The remaining work is to connect the top-level `run` and `test` commands to the native bridge behavior consistently.
- `runtime-ci` still documents the native path we want to standardize on.
- Phase 5 is not complete for the top-level CLI flow in this workspace snapshot; the normal `run`/`test` entrypoints still need migration.
