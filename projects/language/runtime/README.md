# Norscode Runtime

This directory is the beginning of the native Norscode runtime.

Current scope:

- bytecode loader for `norscode-bytecode-v1`
- opcode and operand enums
- runtime value model
- runtime error types
- standard I/O abstraction
- VM skeleton with stack helpers
- builtins split across dedicated runtime modules:
  - `builtins_core` for `argv()`
  - `builtins_text` for text helpers
  - `builtins_list` for mutating list helpers
  - `builtins_gui` for headless `std.gui` compatibility
  - `builtins_web` for `std.web` HTML/CSS/Sass helpers
  - `builtins_source` for the `kjor_kilde()` studio bridge
  - `builtins_fs` for filesystem and command helpers
  - `builtins_assert` for assertion helpers
- cargo-based build target for the native runtime binary
- runtime-build can now also write JSON with the cargo path, binary path, and build output
- comparison bridge for interpreter/runtime parity via `runtime-compare`
- compare-suite bridge for multiple representative interpreter/runtime parity checks
- runtime-compare JSON now also carries top-level `build_output` when auto-build happens
- runtime-compare and runtime-compare-suite now try to build the native runtime automatically before they fail when the default binary is missing; they are strict native and do not fall back to bytecode, and if you pass `--runtime-binary`, that explicit binary is used without fallback
- bytecode-run now also supports JSON with explicit `build_output` and accepts `.no`, `.nast.json` and `.ncb.json`
- bytecode-build now also supports JSON for the generated bytecode path and metadata, and is best treated as a low-level developer tool rather than the primary end-user flow
- build now also supports JSON for the generated C and executable paths
- runtime-test now requires native runtime and does not fall back to bytecode when the default binary is missing; if you pass `--runtime-binary`, that explicit binary is used without fallback
- runtime-test JSON now also carries top-level `build_output` when auto-build happens
- `norscode test` JSON now also carries top-level `build_output` when the suite auto-builds native runtime
- `norscode test`, `runtime-suite`, `runtime-compare-suite`, `runtime-verify`,
  `runtime-smoke`, `runtime-compare-verify`, and `runtime-ci` JSON also mark
  `fallback_used` on the relevant results and `fallback_used_count` in suite
  or CI summaries when fallback is actually used
- `runtime-test` and `norscode test <file> --json` also expose
  `fallback_used_count` on the single-result payload
- `runtime-verify`, `runtime-smoke`, and `runtime-compare-verify` JSON also
  show `fallback_used` and `fallback_used_count` explicitly on the top-level
  payload for that run, and they also expose `runtime_backend` at top level
- `runtime-compare` JSON also marks `fallback_used` on the individual result
  whenever that command actually uses a fallback path
- `norscode run --engine python --json` also exposes `fallback_used` and
  `fallback_used_count` at top level
- `bytecode-run` is now strict native and does not fall back to bytecode when
  native runtime is missing
- `runtime-compare` JSON also exposes `fallback_used_count` on the result
- those same JSON outputs also include `fallback_used_count` and
  `runtime_backend`, so scripts can tell whether the result came from native
  runtime or the bytecode fallback
- the full `norscode test` suite path now is native-only and no longer falls back to bytecode
- the single-file `norscode test <file>` path now is native-only and no longer falls back to bytecode
- runtime-suite and runtime-compare-suite JSON now carry top-level `build_output` when auto-build happens
- runtime-suite and runtime-compare-suite also treat an explicit `--runtime-binary` as strict and do not fall back when that binary is missing
- runtime-ci JSON now also carries top-level `build_output` when auto-build happens
- runtime-verify, runtime-smoke, and runtime-compare-verify JSON also carry top-level `build_output` when auto-build happens; if you pass `--runtime-binary`, that explicit binary is used without fallback
- status bridge that reports whether native runtime and compare-verify are available, whether a native build can start, whether the runtime binary exists, whether fallback is active for the current command, plus the next concrete step, next command, and full doctor summary in JSON
- doctor bridge that reports the current native-runtime status and any optional tooling/export follow-up, and also reports fallback state
